# Admin Front

后台管理系统前端，基于 `Vue 3 + Vite + Element Plus + Pinia + Vue Router`。

这个项目的核心目标不是只做静态后台页，而是支持：

- 登录鉴权
- 动态菜单
- 动态路由
- 按钮级权限控制
- 系统管理与业务管理共存

如果要给同事介绍，这个前端可以理解成一句话：

> 一个围绕“菜单驱动页面、权限驱动按钮”的后台壳子，系统管理和业务页面都挂在这套权限模型上。

## 项目定位

当前前端已经覆盖两类页面：

- 系统管理
  - 用户管理
  - 部门管理
  - 角色管理
  - 菜单管理
  - 数据字典
- 业务占位页面
  - 桥梁管理
  - 隧道管理
  - 附属设施管理
  - 养护站管理

其中系统管理负责“用户、角色、菜单、字典、部门”这些平台能力；  
业务页面负责承接后续真正的业务模块。

## 技术栈

- Vue 3
- Vite
- Element Plus
- Pinia
- Vue Router 4
- Axios

## 目录结构

```text
src/
  api/                    接口封装
  composables/            组合式能力封装
  directives/             全局指令
  layout/                 后台框架布局
  router/                 路由与动态菜单运行时
  stores/                 Pinia 状态管理
  utils/                  请求封装等基础工具
  views/
    system/               系统管理页面
    assetManagement/      业务占位页面
    Dashboard.vue         首页
    Login.vue             登录页
    Profile.vue           个人中心
    NoMenu.vue            未分配菜单提示页
  App.vue
  main.js
```

## 模块说明

### `api`

负责按业务拆分接口调用，不在页面里直接拼 URL。

- `auth.js`
  - 登录
  - 当前用户信息
  - 当前用户菜单
- `user.js`
  - 用户增删改查
  - 分配角色
- `role.js`
  - 角色增删改查
  - 分配菜单
  - 设置数据权限
- `menu.js`
  - 菜单增删改查
- `department.js`
  - 部门树与部门管理
- `dict.js`
  - 字典类型和字典数据

### `utils/request.js`

统一 Axios 封装，作用是：

- 统一 `baseURL`
- 自动带上 JWT Token
- 统一接口报错提示
- 统一返回后端 `response.data`

也就是说，页面层拿到的是已经处理过的响应，不需要反复写重复的鉴权和报错逻辑。

### `stores/user.js`

用户会话中心，负责：

- 保存 token
- 保存当前登录用户信息
- 保存当前用户菜单树
- 登录后初始化用户上下文
- 退出时清空会话

这个 store 是“动态路由”和“按钮权限”的数据来源。

### `router/index.js`

全局路由入口，负责：

- 基础静态路由注册
  - 登录页
  - 布局页
  - 个人中心
  - 未分配菜单页
- 登录状态判断
- 首次进入时动态注册菜单路由
- 无菜单权限时跳转到 `NoMenu`

### `router/menuRuntime.js`

这是前端动态菜单的核心文件，建议重点向同事讲。

它的职责是：

- 规范菜单里的 `path`
- 规范菜单里的 `component`
- 把菜单树转换成可注册的动态路由
- 计算首个可访问页面
- 过滤真正能渲染到左侧菜单里的项目

可以把它理解为：

> 菜单数据到 Vue Router 路由对象之间的运行时转换层。

### `layout`

后台整体布局，包括：

- 左侧菜单
- 顶部栏
- 主内容区

左侧菜单不是写死的，而是根据当前用户菜单树递归渲染。

### `composables/usePermission.js`

按钮权限组合式工具，负责：

- 从当前用户菜单树中提取 `button` 类型权限标识
- 提供 `hasPermission()` 和 `hasAnyPermission()`

适合在页面中做“权限摘要”“权限说明”“组合判断”。

### `directives/permission.js`

全局按钮权限指令，作用是：

- 根据当前用户菜单中的按钮权限，控制元素显示隐藏

典型写法：

```vue
<el-button v-permission="'asset:bridge:create'">新增桥梁</el-button>
```

这样页面开发时就不用每个按钮都手写 `v-if="hasPermission(...)"`。

### `views/system`

系统管理页面集合：

- `User.vue`
  - 用户列表
  - 新增/编辑用户
  - 分配角色
  - 选择所属部门
- `Department.vue`
  - 部门树管理
- `Role.vue`
  - 角色管理
  - 权限配置
  - 数据权限配置入口
- `Menu.vue`
  - 菜单与按钮配置
  - 动态路由数据来源
- `Dict.vue`
  - 字典类型和字典数据维护
- `RouteMissing.vue`
  - 菜单配置了不存在组件时的兜底页

### `views/assetManagement`

业务页面占位模块，当前主要作用有两个：

- 作为动态路由的真实组件承载点
- 演示业务页面如何接入按钮权限

当前桥梁管理页已经接了按钮权限演示。

## 权限与路由链路

这个项目里有两层权限：

### 1. 菜单权限

菜单权限决定：

- 用户能看到哪些左侧菜单
- 用户能进入哪些页面
- 前端会注册哪些动态路由

流程是：

1. 登录成功
2. 拉取当前用户菜单树
3. `menuRuntime` 把菜单树转成动态路由
4. 左侧菜单按菜单树递归渲染

### 2. 按钮权限

按钮权限决定：

- 页面上的新增、编辑、删除、导出等操作按钮是否显示

流程是：

1. 后端把 `button` 类型菜单一并返回给前端
2. 前端从菜单树中提取 `permission`
3. 用 `v-permission` 或 `usePermission()` 做控制

### 3. 未分配菜单处理

当前逻辑已经收紧：

- 如果用户没有分配任何菜单
- 前端不会再退回全量菜单表
- 而是直接进入“未分配菜单”页面

这一步是为了避免菜单越权显示。

## 常见业务页面接入方式

后续新加一个业务页，通常只需要做三步：

1. 在 `views/` 下创建实际页面组件
2. 在菜单管理里配置：
   - `path`
   - `component`
   - 页面按钮权限
3. 给角色勾选对应菜单和按钮

这样前端就能做到：

- 页面自动进入左侧菜单
- 路由自动注册
- 按钮按权限显示

## 本地开发

安装依赖：

```bash
pnpm install
```

启动开发环境：

```bash
pnpm dev
```

构建：

```bash
pnpm build
```

默认本地访问：

- 前端：`http://localhost:5173`

开发代理默认转发到后端 `http://localhost:3101`。

## 总结

> 这个前端不是普通静态后台模板，它的核心是“菜单驱动路由、按钮驱动权限”。系统管理页负责维护用户、角色、菜单、部门和字典；业务页只要在菜单里配置好 path 和 component，就能自动出现在左侧菜单里。按钮权限则通过 `v-permission` 指令控制，做到角色不同、按钮不同。
